# Parameter Golf Diary — April 17-18, 2026

## The Journey: From Baseline to Beating It

This is a learning project. The goal isn't just the number — it's understanding what each technique does and why. Here's the full story of how we went from the unmodified baseline to beating it.

### The Challenge

OpenAI's Parameter Golf: train a language model in **10 minutes on 8×H100**, squeeze it into **16MB**, and minimize val_bpb (bits per byte). The naive baseline scores **1.2244**. The SOTA is **1.0810**. Deadline: April 30, 2026.

We run on **2×H100** (cheaper), so all our experiments use more wall time to compensate.

---

## Phase 0: Getting Our Bearings (Exp 0-5)

### Exp 0-1: Baseline runs
First thing — just run the unmodified code and understand what we're working with.
- 1×H100: 1.3586 val_bpb at ~1200 steps
- 2×H100: **1.2728** val_bpb at ~2975 steps (our reference baseline)

The code is a clean 17M-param GPT with 9 transformer layers, GQA (8 Q heads, 4 KV heads), ReLU² activation, RoPE position encoding, and INT8 quantization for the submission artifact.

### Exp 2: Depth Recurrence — our first modification
**What:** Keep 9 physical layers but replay layers 3 and 4 twice: `[0,1,2,3,4,3,4,5,6,7,8]` = 11 virtual passes with 9 layers of params.

**Why:** Top submissions all use this. More effective depth without more parameters. The model gets to "think twice" through the middle layers.

**Result:** 1.3449 on 1×H100 → **-0.014 vs baseline**. First win! 10% slower per step (11 virtual passes vs 9) but better quality per step.

### Exp 3-4: Confirming on 2×H100
Recurrence on 2×H100: 1.2728 vs baseline 1.2732. Tiny gap (-0.0004) because the slower steps mean fewer total steps in the same wall time. But at matched step counts, recurrence wins by -0.017 at step 2000. The advantage grows with more training.

**Lesson:** Recurrence is a real improvement, but you need enough steps for it to pay off. On short runs the step-time overhead eats the quality gain.

## Phase 1: Hyperparameter Sweep (Exp 7-14)

### Exp 7-9: q_gain sweep
**What:** `q_gain` scales the query vectors before attention. Higher = sharper attention patterns. Top submissions use 5.25 (vs default 1.5).

| q_gain | val_bpb | Δ vs baseline |
|---|---|---|
| 1.5 (default) | 1.2728 | — |
| 3.0 | 1.2713 | -0.002 |
| **4.0** | **1.2693** | **-0.004** |
| 5.25 | 1.2754 | +0.003 |

**Lesson:** 4.0 is our sweet spot, not the 5.25 that top submissions use. Their optimal value depends on their other changes (different model width, different activation, etc). Don't blindly copy hyperparameters — sweep for your config.

### Exp 10: 3-layer recurrence [2,3,4]
Adding a third recurring layer gave 1.2698 (-0.003) vs 2-layer's 1.2693 (-0.004). **Diminishing returns.** Stick with 2 layers.

### Exp 11-12: Model shape experiments
- **Narrow + deep** (dim=448, 11 layers): 1.2817 (+0.009) — WORSE
- **Wide MLP** (mult=3, dim=384): 1.2892 (+0.016) — WORST

**Lesson:** At 17M params, width matters more than depth. Don't shrink model_dim below 512.

### Exp 13: seq_len=2048 — the biggest lever
**What:** Double the training sequence length from 1024 to 2048 tokens.

**Why:** Longer context → model sees full paragraphs instead of fragments → learns structure more efficiently per step.

**Result: 1.2583 (-0.015)** — biggest single improvement we found. Each step processes 2× more tokens but takes roughly the same wall time. RoPE handles arbitrary lengths natively, so it's just an env var change.

### Exp 14: Longer warmdown
WARMDOWN_ITERS=2400 (vs default 1200): 1.2736 (+0.001). **No help.** Don't over-engineer the LR schedule.

## Phase 2: Stacking Structural Changes (Exp 15-16)

### Exp 15: Combined config
Recurrence + seq_len=2048 + q_gain=4.0: **1.2590 (-0.014)**. Gains stack cleanly — consistent -0.005 improvement over Exp 13 at every checkpoint.

### Exp 16: Phase 1 complete — all structural improvements
Added on top of Exp 15:
1. **LeakyReLU(0.5)** instead of ReLU² — gentler gradient flow through negative values
2. **Parallel residuals** from layer 7+ — attention and MLP read from same input (GPT-J style)
3. **Staged recurrence** — train plain for first 1000 steps, then activate recurrence
4. **Partial RoPE** (16/64 dims) — only rotate 16 of 64 head dims, rest are position-free content dims

**Result: 1.2550 (-0.018)** — new best! But the 84-second torch.compile recompilation cost when recurrence activates was painful (14% of a 10-min budget).

**What we learned about each change:**
- LeakyReLU: visible improvement from step 400 onward. Stable, reliable.
- Parallel residuals: no instability, consistent help. GPT-J got this right.
- Staged recurrence: the recompile cost is brutal on short runs. On 2×H100 10min it barely pays for itself.
- Partial RoPE: subtle but helps — gives the model some position-free "content" dimensions.

**Key insight:** Structural changes stack. Each one helps independently and they combine cleanly. This validates the "stable over finetuned" philosophy — these aren't fragile hyperparameter tricks, they're architectural improvements.

## Phase 3: The Full Run (Exp 21)

### The run

**Config:** 2×H100, 40 min, all improvements from Exp 16 + native PyTorch GQA (PyTorch 2.8.0)

**Result: 1.1963 INT8 val_bpb**

### How it unfolded

**Steps 0-1000 (pre-recurrence):** Matched Exp 16 almost exactly. By step 1000: 1.3567 (Exp 16 was 1.3542). Gap of +0.003. Native GQA giving ~210ms/step.

**Step 1000 — recurrence activates.** 77-second torch.compile pause.

**Steps 1200-2200 (both runs compared):** Exp 21 fell behind Exp 16 by an increasing margin (+0.000 at step 1200, +0.026 at step 2200). This was confusing until we realized: Exp 16's warmdown started at step ~1100, so it was already settling. Exp 21 was still at full LR — higher noise, but still learning.

**Step 3600 — beat Exp 16's final** (1.2523 < 1.2550). Uncharted territory.

**Steps 3600-8200 — diminishing returns at full LR.** Slow, steady drop: 1.2523 → 1.2175. About -0.002/checkpoint. The model was extracting less and less from each step at this learning rate.

**Steps ~8300-9511 — warmdown freefall.** THIS was the surprise. The code computes warmdown relative to the wall clock endpoint, not a fixed step count. Warmdown started at ~step 8300 (9511 - 1200), not step 4800 (6000 - 1200) like I expected. Once LR started decaying:

```
Step 8200: 1.2175  ← warmdown begins
Step 8600: 1.2103  (-0.005/checkpoint!)
Step 8800: 1.2054
Step 9000: 1.2008  ← sub-1.20!
Step 9200: 1.1957
Step 9400: 1.1913
Step 9511: 1.1898  ← wall clock cap
INT8:      1.1963  ← final (+0.0065 quant cost)
```

### What each improvement contributed

Working backwards from our best:

| Technique | Estimated contribution | How we know |
|---|---|---|
| seq_len=2048 | -0.015 | Exp 13 vs Exp 5 |
| Depth recurrence | -0.004 | Exp 4 vs Exp 3 (at matched steps) |
| q_gain=4.0 | -0.004 | Exp 9 vs Exp 5 |
| LeakyReLU(0.5) | ~-0.003 | Exp 16 vs Exp 15 (estimated) |
| Parallel residuals | ~-0.002 | Exp 16 vs Exp 15 (estimated) |
| Partial RoPE | ~-0.001 | hard to isolate |
| Longer training (40 vs 10 min) | -0.059 | Exp 21 vs Exp 16 |

The longer training time is by far the biggest factor. Our structural improvements gave ~-0.018 in 10 min. The extra 30 min gave another -0.059. Diminishing returns per step, but there's a LOT of value in just training longer.

## Caveats

**This is NOT a valid submission.** The challenge is 8×H100 for 10 min. We ran 2×H100 for 40 min (same GPU-minutes, different wall clock).

**Projected 8×H100 10min result:** ~1.207. We'd get ~8000 steps (vs our 9511) with a properly-timed warmdown. Still beats the baseline (1.2244) but not by the dramatic 0.028 margin.

**Context length matters for the comparison.** The baseline runs seq_len=1024 and gets ~13780 steps. We run seq_len=2048 and get fewer steps. The community consensus (and top submissions) is that longer context wins, but it's a tradeoff.

**Gap to SOTA is still 0.115.** The top submissions use techniques we haven't implemented: SP8192 tokenizer, U-Net skip connections, Legal TTT (test-time training), full GPTQ quantization, sliding window eval. The tokenizer alone is probably worth 0.03-0.05.

## Phase 4: Running the SOTA (Exp 22-23)

After Exp 21, we decided to stop building our own improvements and try running the actual SOTA submission code. The #1 leaderboard entry (1.0810) is open source in the repo's records directory — obfuscated in an LZMA-compressed blob, but fully functional.

### What the SOTA has that we don't

| Technique | SOTA | Ours |
|---|---|---|
| Model size | 35.9M (11L, MLP 4×) | 17M (9L, MLP 2×) |
| Tokenizer | SP8192 | SP1024 |
| Quantization | INT6 GPTQ + Brotli | INT8 + zlib |
| Optimizer | MuonEq-R (row-normalized) | Muon |
| XSA | all 11 layers | none |
| Skip gates (U-Net) | yes | none |
| EMA | 0.9965 | disabled |
| TTT | score-first SGD | none |
| Warmdown | 72% of training | 20% |

The SOTA is a much bigger model (35.9M vs 17M) that fits in 16MB because INT6 GPTQ compresses ~2× better than INT8.

### Getting SP8192 data

This was the main obstacle. SP8192 data isn't available as a download — we had to retokenize FineWeb from scratch:

1. Download `docs_selected.jsonl` from HuggingFace (~45GB raw text)
2. Copy the SP8192 SentencePiece model from a previous submission's records
3. Run `download_hf_docs_and_tokenize.py` to produce 83 train shards + 1 val shard

This took ~30 min and hit multiple disk quota issues (the 45GB docs file + 16GB SP1024 data + 15GB SP8192 data = ~76GB). We had to resize the network volume from 30GB → 150GB.

The SP8192 data now persists on the network volume — no re-tokenization needed for future runs.

### Exp 22: SOTA with SP1024 (1×H100, 10min)

First test — run the SOTA code with our existing SP1024 data. Just to prove the code works.

**Result:** 534 steps, val_bpb = 1.6050 INT8. Bad score but everything works: training, EMA, GPTQ INT6, Brotli compression, eval. The code is a single obfuscated file that unpacks and runs a 470-line training script.

**Key observation:** Post-EMA val_bpb (1.5976) was worse than the last training val_bpb (1.2929). EMA needs many more steps to be useful — with only 534 steps the averaged weights are mostly the random initialization.

### Exp 23: SOTA with SP8192 (1×H100, 10min)

The real deal — full SOTA architecture with the right tokenizer.

**Result:** 558 steps, val_bpb = 1.4052 INT8. Still bad (need 4550 steps), but proves SP8192 pipeline works end-to-end.

**Timing breakdown on 1×H100:**
- Initial compile: ~1.5 min
- Pre-recurrence training (steps 1-236): ~3 min
- Recurrence recompile: ~4-5 min (17 virtual layers!)
- Post-recurrence training (steps 237-558): ~1.5 min
- GPTQ + eval: ~2 min

The recompile alone eats **half the 10 min budget** on 1×H100. This is why you need 8 GPUs — not just for throughput, but because the compile overhead is the same regardless of GPU count.

**Submission size issue:** 16,005,147 bytes — 5KB over the 16MB limit. The extra train shards (87 vs 80 in the original) or slight weight differences caused this. Minor fix needed.

**Scaling confirmed:** 1×H100 gets 558 steps in 588s. 8×H100 gets 4550 steps in 588s. Ratio: 8.15× — almost exactly the GPU count. Full replication needs ~80 min on 1×H100 or ~40 min on 2×H100.
