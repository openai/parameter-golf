# 2026-04-22 — Specs 017–019b: learnable α full pipeline, throughput arc, 019b miss

---

## Starting point: 016 validated the mechanism

Specs 015 and 016 confirmed that learnable per-pass α blending produces real per-step improvement
(val@4000 −0.0032 vs 008) and that init (zeros vs ones) is irrelevant. The logical next step:
run a full 8×H pipeline (train + GPTQ + TTT) to see if the per-step gain survives quantization
and test-time training.

---

## Spec 017 — First full pipeline with learnable α

017 was the first run to go all the way: 8×H100 JP, learnable fp32 α, full phased TTT.

Result: pre-quant **1.06861**, post-TTT **1.06733**.

At face value this looked like a miss vs #1736 (1.06610) by 0.00123. But the per-step signal
was real — val@4000 was better than 008 by ~0.002. The story coming out of 017 was "good
mechanism, not enough steps."

The α converged to an interesting pattern:
- Pass 2: amplifies with depth [1.078 → 1.273 → 1.430] — layer 5 gets the strongest weight
- Pass 3: damps with depth [1.016 → 0.973 → 0.832] — layer 5 most suppressed

A depth-monotonicity inversion between the two extra passes. The model discovered something
real: pass 2 drives L5 hard, pass 3 pulls it back. Saved these as the "017 endpoint values"
for subsequent frozen-buffer experiments.

**Important caveat (discovered much later):** 017's post-TTT 1.06733 was measured with a TTT
α bug — `forward_ttt` didn't apply the α blend, so TTT was adapting a model that didn't
match training. This wasn't known at the time.

## Specs 018 / 018b / 018c — Throughput diagnostic arc

Three small proxy experiments (2×H100, 6L/256d) exploring blend form options:

**018 — torch.lerp:** Replacing the 4-op manual blend with `torch.lerp(x_before, x_new, α)`
should fuse into a single CUDA primitive. Measured: **+0.6% throughput recovery** — only
18% of the blend overhead recovered. Disappointing. Conclusion: torch.compile was already
partially fusing the manual blend; lerp squeezed the residual but not dramatically.

**018b — bake-in refactor:** Pre-compute carry at the start of the loop, eliminating the
late-blend indirection. Result: worse throughput. Shelved.

**018c — constant-α (compile-time literal):** Replace `nn.Parameter` α with Python float
literals at 017's endpoint values. torch.compile specializes each lerp kernel per-site with
the known constant → **92% of blend overhead recovered**. This is the correct optimization.
With literal constants, Dynamo folds the `1-α` subtraction, bakes the value into the CUDA
kernel, and reformulates surrounding-op fusion. At full 11L/512d the residual overhead
approaches zero.

The upgrade path: spec 019 uses constant-α + torch.lerp at full scale.

---

## Spec 019 — Constant-α full pipeline: unexpected throughput regression

019 launched on 8×H JP with literal constants at 017's endpoint values and torch.lerp.
The 018c proxy predicted near-zero overhead — so we expected ~40-65 more steps than 017.

Result: **4,697 steps**, post-TTT **1.06744**. Actually *fewer* steps than 017's 4,784. The
run ran ~130K tok/s slower throughout — a persistent deficit, not node noise (which is
typically tens of K, not hundreds of K).

The proxy→full extrapolation failed. Constant-α + torch.lerp is genuinely slower at full
scale than 017's tensor-α + manual blend. The mechanism wasn't understood, but at 11L/512d
the Inductor fusion heuristics apparently shift in a way that penalizes torch.lerp's
primitive template.

The per-step quality was still good — val@4000 **1.1071** was better than both 008 (1.1110)
and 017 (1.1088). The model quality signal was real; we just got fewer steps to convert it
into endpoint val_bpb. Step-matched extrapolation to 4,828 steps projects post-TTT ~**1.063**
— a clear beat of #1736 if we could only fix the throughput.

---

## Spec 019b — Algebraic lerp form; OOM fix; closest miss yet

The diagnostic for 019's throughput regression: torch.lerp was the culprit. Switching to
algebraic form `x_before + α*(x_new - x_before)` reduces AoT partitioner overhead.

But the first attempt hit CUDA OOM on all 8 ranks during the initial compile warmup:
`_run_cu_bucket_warmup`. Root cause: the manual form `α*x_new + (1-α)*x_before` creates
4 FX nodes per blend site; AoT partitioner's min-cut saved 2 intermediate tensors
× 6 sites × ~512 MiB = 3–6 GiB of extra activation memory. The algebraic form has 3 ops
and 1 intermediate, so the partitioner saves nothing. Fix: switched to algebraic form.

(There was also a tokenizer path bug on the first clean attempt — DATA_DIR pointed to
the wrong subdirectory. Fixed before launch.)

019b result: **4,716 steps**, pre-quant **1.06951**, post-TTT **1.06628** — miss by **0.00018**.

Step-4000 val_bpb: 1.1071 — identical to 019. Algebraic lerp is a true no-op at the loss
level; the algebraic/manual distinction affects memory layout and compilation, not gradients.
The 19 extra steps (4716 vs 4697) are pod lottery, not the code change.

Throughput drift: same pattern as 019. Starts at ~8.08M tok/s, drifts to ~6.2M by minute 8.
Algebraic form did not fix the throughput root cause.

**0.00018 miss is within single-seed std (~0.0002).** The prescribed next action per spec
was 3-seed to determine if 1.06628 is representative. However, the throughput regression
remained undiagnosed — every run was leaving ~130 steps on the table, which corresponded
to ~0.007 post-TTT bpb.

## Commit-hash confusion incident

A key methodology failure was discovered during the 021c arc (later). The commit we'd been
calling "019b reference" (`e93d77d`) was actually the *pre-OOM-fix manual-add variant* —
not the true 019b-submission code. The real 019b submission is at `9517a3b` (algebraic form).

This meant: when we compared 021 variants against "019b" at 4×H, we were comparing against
the wrong baseline. Arm A' was added in spec 021c specifically to correct this. It ran
`9517a3b` and showed algebraic beats manual-add by ~0.0005 at 4×H — a real and reproducible
effect.

Lesson: always verify commit hashes against submission artifacts before citing a number as
the reference. "019b commit" was ambiguous between two different code paths.
