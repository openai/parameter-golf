# 2026-04-22 — Specs 021 → 021(fix) → 021b: buffer hypothesis, α-value bug, bf16 dtype

---

## Context: why register_buffer?

After 019/019b, the throughput drift (8.0M → 6.2M tok/s over 8 minutes) was undiagnosed.
A diagnostic run (spec 020b) had identified "Type B mystery spikes" — sudden 30–50% tok/s
drops at irregular steps — as a distinct pattern in the learnable-α runs. Hypothesis: tensor
α is a `nn.Parameter`, which Dynamo must specialize around at every recompile trigger. Each
post-validation recompile creates a new graph specialization for the current α values, causing
compile stalls and the choppy throughput profile.

Spec 021's fix: promote α from `nn.Parameter` to `register_buffer`, frozen at 017's endpoint
values. A buffer is not a parameter; Dynamo doesn't specialize on it dynamically. This should
kill the Type B spikes and the post-validation recompile cluster, giving clean throughput.

---

## Spec 021 — Buffer-α in production

First 8×H100 JP run with α as `register_buffer` at 017's endpoint values.

Result (commit `cb5cd78`):
- 4,883 steps — **186 more than 019**, the cleanest step count yet
- Pre-quant EMA: **1.06963**
- Post-TTT: **1.06900**

Throughput hypothesis confirmed: **0 Type B spikes**, 0 post-val recompile clusters.
Pre-loop plateau at 8.13M tok/s. Clean, flat profile throughout. This was a real win for
training stability.

But the post-TTT headline (1.06900 vs 019b's 1.06628) was a significant regression — 0.0027
worse despite better throughput and more steps. The TTT step was the culprit:

| run | post-GPTQ | post-TTT | TTT delta |
|---|---|---|---|
| 019 (literal-α) | 1.07989 | 1.06744 | −0.01245 |
| 021 (buffer-α) | 1.07913 | 1.06900 | **−0.01013** |

021's TTT only recovered 0.0101 vs 019's 0.0125. The hypothesis: frozen α in a register_buffer
means TTT's LoRA adapters cannot modulate the blend coefficients during per-document adaptation.
They have to route all corrections through projection weights alone. 019's literal-α got the
same frozen treatment structurally, but Dynamo/Inductor may have inlined the constants
differently, leaving more rewrite flexibility for TTT. Either way, buffer-α trades TTT
headroom for throughput stability.

**However — this finding was immediately flagged as provisional.** A careful check of the
hardcoded buffer values found a bug.

---

## The α-value bug

The `cb5cd78` buffer tensor hardcoded pass-3 L4 α as **0.96484375**. The actual 017 endpoint
value at that site is **0.97265625** — one 1/128 tick higher. (All α values are stored in
bf16, which has 1/128 resolution in the [0.5, 1.0] range.)

The training-loss divergence from 019b began at exactly the loop-activation step — consistent
with the bug being a contributor. The fix-run (`dc0b5f8`) was launched immediately on the
same pod with the corrected value.

Fix-run result: pre-quant **1.06950**, post-TTT **1.06900**.

The one-tick correction moved pre-quant by +0.00013 — essentially zero. **The α-value bug
was not the root cause.** 021's TTT regression vs 019b persists with the correct values.
The buffer-α → reduced TTT flexibility story stands.

---

## Spec 021b — bf16 dtype for the buffer

New hypothesis: the fix-run's +0.00072 pre-quant improvement over the original 021 (1.06878
vs 1.06950) came from something other than the bug. Looking at the code: the original 021
stored the buffer in float32 and cast `.to(x_new.dtype)` at each blend site. The cast
happens inside the compiled graph and may prevent Inductor from fusing the cast with the
lerp. If the buffer is stored as bf16 and the runtime cast removed, the graph is cleaner.

Result (commit `d070df3`): pre-quant **1.06878**, best of the 021 family at 8×H at this
point.

Δ vs 021-fix: **−0.00072** — closes about half the gap to 019b (1.06951 - 1.06878 = 0.00073
still remaining). Consistent with dtype/fusion being *a* contributor to 021's regression
but not the full story. Post-TTT not measured (pod stopped after GPTQ to keep within budget).

---

## State entering the container A/B/C test

The 021 arc to this point established:
1. Buffer-α kills Type B spikes and post-val recompiles — real throughput win
2. α-value typo: irrelevant (+0.00013)
3. bf16 dtype alignment: closes ~half the gap to 019b
4. Residual gap: still not explained — must be something else in the graph

Two remaining hypotheses: (a) `register_buffer` vs `nn.Parameter(requires_grad=False)` has
different Dynamo behavior even for frozen tensors; (b) the manual-add blend form in 021/021b
differs from 019b's algebraic form in bf16. These are tested in spec 021c.
