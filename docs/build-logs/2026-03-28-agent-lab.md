## For readers new to the pieces

**Parameter Golf** asks: train a small language model under a **size** budget (code + compressed weights ≤ 16 MB) and **time** budgets on the official hardware, and score it by how well it **compresses** held-out text — reported as **`val_bpb`** (validation bits per byte). Lower is better: fewer bits per byte means the model assigns higher probability to the real text.

**GQA (grouped-query attention)** — Instead of every attention head having its own **K** and **V** projections, several **query** heads share one **KV** pair. **`NUM_KV_HEADS`** controls how many KV groups exist; fewer KV heads usually mean **less memory** and **less work** per token, which can allow **more training steps** in the same wall-clock cap.

**Global batch in tokens** — `TRAIN_BATCH_TOKENS` is how many tokens you process per **optimizer step** (after micro-batching). **Halving** it often means **more steps** in the same wall time if each step gets cheaper — a trade-off between noise and update count.

**Muon** — An optimizer variant used here for matrix-shaped weights. **`MATRIX_LR`** is the learning rate for those updates; too low can mean slow progress; too high can destabilize.

## Official challenge: two separate time limits

The README states that **record** submissions must:

1. Finish **training** in **~≤10 minutes on 8× H100**, and  
2. Finish **evaluation** in **~≤10 minutes on 8× H100** as well — **on top of** training, not merged into one 10-minute window.

So: **yes**, evaluation is also capped for the **leaderboard** setting, on **8× H100**.

Our **local** runs used a **single RTX 3090** and spent **~10 minutes** on the training loop (`MAX_WALLCLOCK_SECONDS=600`) but **~20+ minutes** on the **test-time LoRA** evaluation (`final_int8_ttt_lora`). That does **not** mean the submission is valid for the record track until we verify the **full** train+eval pipeline on the official hardware within **both** caps. See [`agent_lab/CHALLENGE_TIMELIMITS.md`](../../agent_lab/CHALLENGE_TIMELIMITS.md).

## Experiment AL-20260328-001 — Baseline (no code change)

**What we wanted to know:** What score does the **stock** `agent_lab/train_gpt.py` get on **one training shard** with a **10-minute** training wall?

**What happened:** Training stopped at the wall clock at **step 342**. The mid-train validation line showed **`val_bpb` ~1.59**. After training, the script runs **quantization**, **zlib roundtrip eval**, then a long **`eval_val_ttt_lora`** pass. The line we care about for the loop:

- **`final_int8_ttt_lora`** → **`val_bpb` 1.6099**

**Reflection:** I was surprised by how long **TTT** took compared to training — **~23 minutes** of eval time on this GPU vs **~10 minutes** of training. So “one experiment” is **not** 15 minutes end-to-end here; it’s closer to **~35–40 minutes** if you wait for the full metric. **That’s a process insight:** for fast iteration, you might later use a **proxy metric** (e.g. zlib roundtrip only) for **screening**, and reserve **TTT** for **confirming** winners — but only if you’re explicit about it so you don’t optimize the wrong number.

**Commit:** **`df152a4`** (harness baseline; no hyperparameter change yet).

## Experiment AL-20260328-002 — Fewer KV heads (GQA)

**Hypothesis (intuition):** With **4 KV heads**, the model is doing more KV work than with **2 KV heads** at the same query head count. If each step is a bit cheaper, we might fit **more steps** in the same **600 seconds** of training — and more steps might mean better loss.

**What we changed in code** (default in `Hyperparameters`):

```python
# Before (typical baseline)
num_kv_heads = int(os.environ.get("NUM_KV_HEADS", 4))

# After (landed in agent_lab)
num_kv_heads = int(os.environ.get("NUM_KV_HEADS", 2))
```

**Result:** **`final_int8_ttt_lora` 1.5921** (better than 1.6099). Step count at wall: **356** vs **342**.

**Verdict:** Hypothesis **correct** for this stack.

**Commit:** **`d9ca661`**

## Experiment AL-20260328-003 — Half global batch (`TRAIN_BATCH_TOKENS`)

**Hypothesis:** If we **halve** tokens per step, each step may be cheaper; in the **same** 600 seconds we might run **many more** optimizer steps. More updates might beat the noise from a smaller batch.

**What we changed:**

```python
# Before
train_batch_tokens = int(os.environ.get("TRAIN_BATCH_TOKENS", 524_288))

# After
train_batch_tokens = int(os.environ.get("TRAIN_BATCH_TOKENS", 262_144))
```

**Result:** **695** steps in 600s; **`final_int8_ttt_lora` 1.4239** — a large jump. VRAM dropped a lot (~**6 GB** peak vs ~**12 GB**).

**Verdict:** Hypothesis **correct** — the step-count win dominated.

**Commit:** **`262d70f`**

---

## Experiment AL-20260328-004 — Higher `MATRIX_LR`

**Hypothesis:** At **0.04**, Muon matrix updates might be **too cautious**; **0.06** gives a bit more movement per step in the same 600s window.

**What we changed:**

```python
# Before
matrix_lr = float(os.environ.get("MATRIX_LR", 0.04))

# After
matrix_lr = float(os.environ.get("MATRIX_LR", 0.06))
```

**Result:** **`final_int8_ttt_lora` 1.4129** (vs 1.4239 on the parent).

**Verdict:** Hypothesis **correct** — modest but real improvement.

**Commit:** **`bab9bf7`**

## What went right / wrong (meta)

- **Right:** Changing **one thing at a time** (after baseline) made it easy to see **KV** and **batch** as the big levers.  
- **Right:** Logging **`final_int8_ttt_lora`** consistently.  
- **Risk:** Local **TTT** wall time is huge; **record** eval must also be **≤ ~10 min on 8×H100** — we need a **submission checklist** before claiming a leaderboard result.  
- **Wrong assumption:** “10 minutes per experiment” — only **training** was 10 minutes; **full** metric included **long eval**.
