- conclusion: the extra depth was not wrong, it was step-starved; giving it more optimizer steps turns it into a clear win on this stack
- Frontier follow-up result:
- `AL-20260329-004` kept `10` layers and reduced `TRAIN_BATCH_TOKENS` further to `131072`

- conclusion: the frontier is flattening; much more step count only bought a tiny gain, and artifact headroom is now almost gone
- Alternate speed-recovery mechanism result:
- `AL-20260329-005` kept `10` layers and `196608` batch but reduced `NUM_KV_HEADS` from `2` to `1`

- `Total submission size int8+zlib: 13399103 bytes`
- `final_int8_zlib_roundtrip_exact val_bpb: 1.43192919`
- conclusion: cheaper attention recovered some speed and artifact headroom, but the quality hit was worse than the best `kv2` depth branches

## New Tranche Design

- Started tranche `T-20260329-B`, the architecture necessity audit, with a first 5-run set focused on `MLP_MULT` versus `NUM_LAYERS`.
- The simple question is: now that extra depth can win, are we under-layered, or are we spending too much of the budget inside each block's MLP?
- The five planned runs are:
- `B1-E1`: replay the clean `10L / MLP2 / 196608 / kv2` anchor
- `B1-E2`: `11L / MLP1 / 196608 / kv2`
- `B1-E3`: `9L / MLP3 / 196608 / kv2`
- `B1-E4`: `8L / MLP3 / 196608 / kv2`
- `B1-E5`: `9L / MLP3 / 131072 / kv2`
- This set should tell us something simple and important:
- if deeper-thinner wins, the current MLP may be oversized
- if shallower-wider wins, width is the better next spend
- if width only works with smaller batch, then width may have the same step-starvation pattern that depth had
