Yes. Raw models usually do **not** discover a good search space from a program alone.

That is the main mistake.

They see source code.
You need them to see:

* bottlenecks
* invariants
* allowed intervention points
* mutation operators
* evaluation ladder
* prior successful edits

So the operational answer is:

**do not ask the model to search the space.
build a space it can search.**

## The core shift

For any program, the search space is not “all possible edits.”

It is:

**the set of plausible interventions on the program’s bottlenecks, under its invariants, scored by a cheap evaluator.**

That is the object.

So you operationalize it in 5 layers.

---

## 1. Build a program-specific ontology of change

Do not start from code tokens.
Start from **classes of useful change**.

For almost any program, mutations fall into a few buckets:

* algorithm change
* data structure change
* caching / memoization
* batching
* pruning / approximation
* loop reordering
* parallelism / async
* memory layout
* precision / numerics
* scheduling / control policy
* heuristic thresholds
* API / library substitution
* precompute vs compute-on-demand
* pipeline restructuring

This is the first thing GPT/Codex was missing.

JSON schemas are too low-level if they force the model to emit concrete config too early.
What you want is a **change grammar**.

Not:

```json
{"lr": 0.001, "beta1": 0.9}
```

But:

* “introduce cache at function boundary X”
* “replace exact step with approximate top-k”
* “fuse stages A and B”
* “promote repeated expensive computation into precomputed index”
* “switch controller from fixed schedule to signal-triggered schedule”

That is a real search space.

---

## 2. Separate invariants from freedoms

A program cannot be optimized unless the system knows what must not break.

So for any codebase, explicitly define:

### Invariants

Things that must remain true.

* output correctness tests
* numerical tolerance
* API behavior
* resource limits
* determinism requirements
* security / safety constraints

### Freedoms

Things the search can change.

* internal algorithms
* cache policy
* batching
* scheduling
* thresholds
* approximate methods
* data layout
* optimizer / trainer logic
* eval cadence

Without this split, models either:

* stay too conservative
* or mutate into junk

This one step makes search much more intelligent.

---

## 3. Discover bottlenecks before proposing mutations

The model should not invent hypotheses from source alone.

It should first consume:

* profiles
* traces
* flamegraphs
* memory breakdowns
* failure cases
* unstable runs
* slow tests
* regression deltas
* accuracy/latency tradeoff curves

Because hypotheses should come from **friction**.

For any given program, ask:

* where is time going?
* where is memory going?
* where does instability start?
* where do outputs degrade?
* where is work repeated?
* where is control flow brittle?
* where are there hard thresholds?
* what grows worst with scale?

That gives you the first search frontier.

So the real pipeline is:

**observe → localize bottleneck → map to mutation class → generate variants**

not

**read code → magically invent improvements**

---

## 4. Use a hierarchical mutation system

You need three mutation scales.

### Level 1: local parameter / threshold edits

Cheap. Frequent.

* constants
* schedules
* branching thresholds
* retry counts
* cache sizes
* regularization weights

### Level 2: operator swaps

Medium risk.

* algorithm A → B
* exact → approximate
* sync → async
* recompute → cache
* dense → sparse
* fixed schedule → adaptive schedule

### Level 3: structural rewrites

Rare. Expensive.

* pipeline rearrangement
* decomposition changes
* controller redesign
* optimizer update rule rewrite
* moving logic across modules

Most mutations should be level 1 and 2.
Level 3 should be rare and archive-driven.

This is how you stop the system from becoming either trivial tuning or total chaos.

---

## 5. Search over patches, not schemas

This is probably your practical answer.

Codex overconstrained itself because it was forced to think in JSON.
That makes sense for config sweeps.
It is bad for method search.

The search unit should be:

**patch + metadata + evaluation result**

Meaning:

* unified diff or code patch
* what mutation class it belongs to
* expected effect
* risk level
* observed metrics

The patch is the phenotype.
The metadata is for learning what kinds of edits work.

So use JSON only as a **log format**, not as the creative substrate.

Let the model write:

* patch
* short rationale
* expected win
* fallback if it fails

Then your harness runs tests and logs outcomes.

That is much closer to how useful search happens.

---

## So how do you “get” the search space?

You do not get it from one brainstorm.

You build it from 4 sources.

### A. Universal operator library

These are general-purpose mutation types that help many programs.
Examples:

* cache repeated calls
* vectorize loop
* batch requests
* early exit
* approximate expensive scoring
* split hot/cold paths
* hoist invariant computation
* asynchronous prefetch
* adaptive schedule
* mixed precision
* branch simplification

This is your prior.

### B. Program decomposition

Break the program into roles:

* ingest
* transform
* search
* score
* aggregate
* decide
* write

Then attach likely mutation families to each role.

This makes the space program-specific.

### C. Runtime evidence

Profiles, failures, regressions.
This tells you where to search first.

### D. Evolutionary lineage

After enough trials, look at what kinds of edits keep surviving.
That becomes your learned search prior.

This part matters a lot:
**the search space is not fixed. It is learned from successful mutations.**

That is how you go from blind search to intelligent search.

## The right architecture

Use the model in three different roles.

### 1. Analyst

Looks at profiles, traces, failing cases.
Outputs:

* bottleneck hypothesis
* likely intervention classes
* risk assessment

### 2. Mutator

Writes a small patch.
Not 12 ideas. One patch.

### 3. Critic

Explains why the patch may fail:

* violates invariant
* unlikely to help hotspot
* benchmark artifact
* too broad
* complexity cost too high

Do not use one model prompt to do all three.
That is where it gets vague and mushy.

---

## The outer loop should look like this

For any given program:

### Step 1

Baseline run:

* collect metrics
* collect traces
* collect failure modes

### Step 2

Decompose program into optimization surfaces:

* compute hotspots
* memory hotspots
* quality bottlenecks
* stability bottlenecks
* control bottlenecks

### Step 3

Map each surface to a mutation library

Example:

* compute hotspot → batching, caching, approximation, vectorization
* instability hotspot → schedule, clipping, damping, controller logic
* memory hotspot → checkpointing, layout, precision, recompute/cache tradeoff

### Step 4

Generate small candidate patches

### Step 5

Run evaluation ladder:

* compile
* unit tests
* smoke benchmark
* short benchmark
* held-out benchmark
* only then long run

### Step 6

Archive results by mutation family, not just best score

You want to know:

* which families keep helping
* where they help
* when they stop helping

### Step 7

Use archive to bias next proposals

That is how the search space sharpens.

## The deeper answer to your frustration

You are expecting the model to do two hard things at once:

1. infer the latent bottleneck structure of a codebase
2. propose meaningful interventions over it

That is too much from raw source.

Humans do not do that either.
They use:

* prior patterns
* profiling
* decomposition
* analogy
* repeated trial

So your system needs the same scaffolding.

## For “optimizing any given program,” the generic template is:

### Representation

Program as:

* code
* tests
* metrics
* traces
* modules
* dependency graph
* past edits

### Search object

Patch over program

### Mutation language

Operator classes, not raw tokens

### Fitness

Task metric plus cost metric plus invariants

### Search policy

Mostly small edits, some medium edits, rare structural jumps

### Memory

Archive of surviving lineages and mutation families

That is enough to generalize across many programs.

## The most important practical design choice

Start with **bottleneck-conditioned mutation templates**.

For example:

If hotspot = repeated expensive pure function
→ try memoization, caching, precompute, hoisting

If hotspot = large candidate scoring
→ try pruning, approximate search, top-k gating, staged scoring

If hotspot = unstable optimization
→ try schedule adaptation, clipping change, trust-region-like damping, optimizer switch trigger

If hotspot = high I/O overhead
→ try batching, prefetch, async, write coalescing

This is much better than “propose improvements.”

It turns search into something operational.

## What I would tell Codex / GPT instead

Not:
“Figure out the search space.”

But:

“Given these traces and tests, identify the top 3 bottlenecks. For each bottleneck, choose from this mutation library the 5 most plausible intervention classes. Produce one minimal patch that targets one bottleneck only. Keep behavior invariant under these tests. Return a diff and a short expected-effect note.”

That prompt is much closer to solvable.

## Minimal viable version

If you want a practical first version:

1. Take one program.
2. Collect traces and metrics.
3. Write a mutation library of 20–40 operator types.
4. Tag each operator by when it is applicable.
5. Let the model choose one operator conditioned on one bottleneck.
6. Have it emit a patch, not JSON config.
7. Run staged evaluation.
8. Store all outcomes in an archive.
9. Periodically mine the archive for successful operator families.

That is enough to start discovering the search space instead of pretending it already exists.

## The short truth

No, the models usually do not just “figure out the search space.”

You get there by building:

* an ontology of interventions
* a bottleneck detector
* a patch-based mutation loop
* an archive that learns which mutation families work

So the search space is not discovered in one shot.

It is **bootstrapped**.

And for optimizing any program, that is the right abstraction:
**search over intervention operators conditioned on observed bottlenecks, with evolution over successful lineages.**
