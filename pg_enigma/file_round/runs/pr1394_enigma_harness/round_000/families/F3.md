# F3 - Optimizer phase-law controller

## Title

Muon/Adam phase-law family.

## Mechanism family

Training-time controller family that changes optimizer geometry, parameter-group behavior, or phase-dependent momentum law before export.

## Search level

Patch-level family.

## Operator family

Momentum warmup rewrites, row-normalization policy changes, and parameter-group repartitioning.

## Target surface

`Muon.step(...)`, `Optimizers(...)`, Muon momentum warmup, row normalization, and matrix/scalar/token parameter grouping.

## Broken predicate

The current MuonEq-R schedule and parameter-group split are already optimal for deployed sliding score under the same wallclock.

## Why baseline truth is false

The optimizer law is hard-coded and clearly consequential for training speed and checkpoint quality, but the deployed objective is still measured only after EMA, GPTQ export, and sliding eval. The baseline therefore reflects one chosen phase law, not a proven optimum.

## Score path trace

Optimizer phase law -> training efficiency and learned state -> EMA checkpoint -> default export -> deserialize -> `quantized_sliding_window`.

## Mutation grammar

`momentum_law := {current_linear_warmup, alternate warmup curve}`  
`row_normalize := {always_on, phase-gated}`  
`group_split := {current split, one alternate split}`

## Cheap falsifier

If matched-wallclock tok/s, train-loss slope, and pre-quant `val_bpb` do not move together in a favorable direction, downstream promotion is already unlikely.

## Preserved contracts

Same architecture, same recurrence program, same export path, same scorer, and no mixed family changes.
