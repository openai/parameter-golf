# F7 - Export reserve / calibration scheduler

## Title

Wallclock partition family.

## Mechanism family

Harness-level family that changes how the training wallclock is partitioned against GPTQ reserve time and calibration effort.

## Search level

Harness-level family.

## Operator family

Reserve-seconds rewrites, calibration-batch budget rewrites, and train-vs-export time partitioning.

## Target surface

`gptq_reserve_seconds`, the wallclock carveout before training, and the calibration-batch budget consumed in `collect_hessians(...)`.

## Broken predicate

The fixed export reserve and calibration budget are already a safe and fair contract for all export-heavy candidates.

## Why baseline truth is false

The findings file shows export-heavy lanes dying in calibration because train and export costs are being paid from one shared budget. So the current reserve rule is not a universal truth about artifact quality.

## Score path trace

Wallclock partition -> available train time and export completion probability -> whether quantized artifact is produced at all -> downstream score only if export finishes.

## Mutation grammar

`reserve_seconds := {smaller, current, larger}`  
`calibration_budget := {reduced, current, expanded}`  
`partition_rule := {fixed carveout, lane-specific carveout}`

## Cheap falsifier

If a candidate only looks better because it changes how much training happened before export, then the family has not isolated artifact quality and should not be ranked against model families.

## Preserved contracts

Same model code, same scorer, same size cap, and no score-semantic changes.

## Verdict

DROP

## Candidate budget

0

## Family compile instructions

Treat this as a harness-policy issue, not a patch-slate family, and do not spend patch-stage slots on it this round. Use export-only bakeoffs instead of reserve hacking when the goal is to compare artifact quality.

## Likely failure mode

The family confounds training budget with export quality and produces rankings that say more about schedule allocation than about the deployed artifact.
