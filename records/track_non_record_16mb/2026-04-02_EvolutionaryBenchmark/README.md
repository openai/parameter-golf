# Evolutionary Transformer Benchmark

This record folder is a RunPod-friendly shell around the evolutionary benchmark tooling in the repo root.

A concise readout of what failed, what worked, and what is still open lives in [RESULTS.md](/Users/kennethmalloy/Local%20Documents/Developer/parameter-golf/records/track_non_record_16mb/2026-04-02_EvolutionaryBenchmark/RESULTS.md).

The goal is not to crown a winner in one run. The goal is to produce a lot of structured evidence before we make tighter decisions about crossover operators, mutation rates, and population size.

## What This Queues

The queue is split into stages:

- `0-throughput`: population evaluation scaling with `vmap`
- `1-viability`: does crossover produce offspring between the parents or garbage
- `2-evolution-core`: first end-to-end loops on the main crossover families
- `3-evolution-sweeps`: mutation, selection, pretrain, and population sweeps
- `4-evolution-seeds`: repeat the most informative operating points with extra seeds

The broad experiment catalog is generated by [generate_evolutionary_matrix.py](/Users/kennethmalloy/Local%20Documents/Developer/parameter-golf/tools/generate_evolutionary_matrix.py), executed by [run_evolutionary_matrix.py](/Users/kennethmalloy/Local%20Documents/Developer/parameter-golf/tools/run_evolutionary_matrix.py), and summarized by [summarize_evolutionary_runs.py](/Users/kennethmalloy/Local%20Documents/Developer/parameter-golf/tools/summarize_evolutionary_runs.py).

## Fast Start

```bash
./runpod_preflight.sh
./runpod_queue_stage0_core.sh
./runpod_queue_stage1_core.sh
./runpod_queue_stage2_core.sh
./runpod_summarize.sh
```

If the pod has enough GPUs and time left, keep going:

```bash
./runpod_queue_stage3_sweeps.sh
./runpod_queue_stage4_seeds.sh
./runpod_summarize.sh
```

To just let it rip:

```bash
./runpod_queue_full.sh
```

For the tighter follow-up batch after the first crossover failures, use:

```bash
./runpod_focused_followup.sh
```

That script does two things the broad queue does not yet emphasize enough:

- chunked bf16 throughput at population sizes that previously OOMed
- a mutation-only `parent_copy` baseline against a matched `delta_overlap` probe
- final-population ensemble scoring for the same probes so we can compare best-individual vs committee behavior

For the committee-first follow-up, use:

```bash
./runpod_committee_scaling.sh
```

That wrapper skips crossover entirely, trains branches with `--member-train-mode parallel_vmap`, and measures how ensemble gains scale with branch count and per-branch train time.

To compare width-vs-depth at a fixed total branch budget, use:

```bash
./runpod_committee_budget480.sh
```

That wrapper keeps total branch-seconds near `480` and varies the split across `4x120`, `6x80`, `12x40`, and `24x20`, which is a much cleaner precursor to larger multi-GPU sweeps.

To test "train deep first, then widen late" at the same total branch budget, use:

```bash
./runpod_committee_schedule480.sh
```

That wrapper runs staged committee schedules such as `4x90->8x15`, `4x60->8x30`, and `4x60->8x15->16x7.5`, which lets us answer whether late breadth can beat the current pure-depth frontier without spending more total branch compute.

To test true funnel schedules, where the committee widens and then prunes back down for another depth phase, use:

```bash
./runpod_committee_funnel480.sh
```

That wrapper keeps total branch-seconds near `480` while trying patterns like `2x90->4x30->8x15->4x15` and `4x60->8x15->4x30`, which is the direct test of whether "narrow wins again" after a useful burst of breadth.

When we have multi-GPU hardware and want to parallelize the most informative committee points instead of the older crossover matrix, use:

```bash
./runpod_queue_committee_frontier.sh
```

That queue stage shards the current committee frontier across visible GPUs. It includes the strongest pure depth-first baselines (`4x120`, `6x80`, `8x60`) and the staged depth-then-breadth schedules (`4x90->8x15`, `4x60->8x30`, `4x60->8x15->16x7.5`) so an `8x H100` pod can turn credits into a lot of committee data quickly.

The current thinking on how to use bigger hardware for committee search versus final committee inference is written up in [MULTI_GPU_PLAN.md](/Users/kennethmalloy/Local%20Documents/Developer/parameter-golf/records/track_non_record_16mb/2026-04-02_EvolutionaryBenchmark/MULTI_GPU_PLAN.md).

To test branch-strength upgrades on top of the current `4x120` committee frontier, use:

```bash
./runpod_committee_base_frontier.sh
```

That wrapper compares the current base against `QK-gain 4.0`, `XSA`, and the combined `XSA + QK-gain 4.0` recipe at the same committee shape. It is the fastest way to ask whether stronger branches move the frontier more than more elaborate late-breadth schedules.

To let the benchmark decide when to widen or narrow based on archive hits, replay disagreement, ensemble gain, and pairwise branch distance, use:

```bash
./runpod_committee_adaptive_frontier.sh
```

That wrapper starts narrow, watches the committee signals after each round, and only widens when the branches are both disagreeing usefully and actually occupying distinct weight territory. It is the cleanest next step beyond hand-written schedules.

To compare the winning byte-level recipe against a real larger-vocab tokenizer instead of just changing `--vocab-size`, use:

```bash
./runpod_committee_vocab_frontier.sh
```

That wrapper keeps the current best `2x120->4x30->8x15` staged schedule and `XSA + QK-gain 4.0` branch recipe, then runs it once with byte tokens and once with the checked-in `sp_bpe_1024` SentencePiece model. It is the right first test for whether larger vocab buys enough efficiency per step to offset the larger embedding table.

To test the “recipe genes, not weight genes” hypothesis directly, use:

```bash
./runpod_recipe_genome_probe.sh
```

That wrapper evolves hyperparameter and architecture recipes under a hard `16MB` artifact budget, using short-train validation BPB as fitness and a longer confirmation pass on the top recipes. It is the cleanest way to ask whether gene-wise crossover in recipe space is a better search primitive than crossover in weight space.

To test whether the winning committee can actually be compressed into a `16MB` artifact as `base + branch deltas`, use:

```bash
./runpod_committee_compressibility.sh
```

That wrapper trains the current best committee recipe, extracts branch deltas from the shared base, and then scores how much ensemble quality survives under three compression families: independent sparse branch deltas, shared union sparse support, and shared low-rank basis on a shared support. This is the direct submission-shape test for “compressed committee” as the artifact.

## What To Look For

The main questions are:

1. Can the H100 evaluate very large populations fast enough for search to be realistic
2. Which crossover families produce offspring between the parents instead of collapse
3. Which evolutionary loops actually improve `val_bpb` over the gradient-trained base
4. Whether the better operators are robust across seeds

The summary script will surface:

- throughput: `models_per_second`, `batch_eval_s`, and peak memory
- viability: fraction between parents, fraction collapse, fraction improving the best parent
- evolution: base vs final `val_bpb`, `test_bpb`, and gain over base
